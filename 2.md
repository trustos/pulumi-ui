Thought for 15s

Yes — the fix is usually to stop making the UI resource-first.

Most end users do not think in terms like “I need a VCN, two route tables, three NSGs, a NAT gateway, and an instance.” They think in goals like:

“I want a public web app”
“I need a private Postgres”
“I want a Nomad cluster”
“I need a VM behind a load balancer”

Pulumi Automation API is specifically meant for building custom cloud interfaces on top of Pulumi, so your UI should lean into that instead of exposing raw IaC primitives first.

The best approach: 4-layer UX

1. Start with intent, not resources

First screen should ask something like:

What are you trying to build?
Which cloud?
Public or private?
Single VM, app platform, DB, Kubernetes, Nomad cluster, networking base, etc.?

So instead of a blank canvas of resources, give them:

Use-case cards
Architecture recipes
guided questions

Examples:

“Deploy a public app”
“Create a private network foundation”
“Run a containerized backend”
“Create a VM with SSH access”
“Provision a Nomad cluster”

This reduces the “I don’t know what resources I need” problem immediately.

2. Internally map each intent to a component

Pulumi already supports Component Resources, which are reusable higher-level building blocks composed of multiple low-level resources. That is exactly the right abstraction for your UI. Instead of exposing 15 OCI/AWS resources, expose one higher-level block like:

NetworkFoundation
PublicWebApp
PrivateDatabase
NomadCluster
TraefikEdge
BastionHost

Then your component expands internally into all required underlying resources. Pulumi explicitly supports authoring and packaging these reusable components.

That means the UI can say:

“Public Web App”
creates: VPC/VCN, subnet(s), security rules, compute/container service, LB, DNS hooks

instead of forcing the user to discover those manually.

3. Add progressive disclosure

Give the beginner a simple mode, and the expert an advanced mode.

For example:

Simple mode

App name
Region
Cloud provider
Public/private
Size
Domain
HA on/off

Advanced mode

exact instance shapes
subnet CIDRs
custom NSGs
route tables
tags
provider-specific overrides
lifecycle options

So the default path is friendly, but power users can still go deep.

4. Generate and explain the draft architecture

After the user answers a few questions, show:

a visual architecture graph
the actual resource list that will be created
estimated monthly cost range
important assumptions
warnings/dependencies

Example:

Based on your choices, we’ll create:

1 VCN
2 subnets
1 internet gateway
1 NAT gateway
2 route tables
3 NSGs
3 compute nodes
1 load balancer

And crucially:

“Why these resources?”

NAT gateway: needed for private nodes to reach the internet for package/image pulls
public subnet: needed for ingress
NSGs: needed to isolate SSH, app traffic, and internal communication

That explanation layer is what builds user trust.

Your entire system should revolve around a blueprint definition, not raw resources.

A good blueprint object might look like this:

type Blueprint = {
id: string;
title: string;
category: string;
description: string;
supportedProviders: string[];
difficulty: "easy" | "medium" | "advanced";
presets: Preset[];
questions: Question[];
outputs: OutputDefinition[];
generatePlan: (answers: WizardAnswers) => ArchitecturePlan;
};

And then:

type ArchitecturePlan = {
components: ComponentPlan[];
resources: ResourcePlan[];
outputs: PlanOutput[];
warnings: string[];
estimatedCost?: CostEstimate;
};

So the user never directly builds resources first.
They fill answers, and the blueprint resolves them into components/resources.
